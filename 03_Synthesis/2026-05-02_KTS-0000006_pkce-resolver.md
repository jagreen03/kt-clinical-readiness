# KTS-0000006: PKCE Resolver Implementation (A-#2)

## 1. Why a Custom Resolver? (The Confidential Client Question)

Spring Security 5.2+ enables PKCE automatically for *public* clients (those without a `client_secret` configured, such as a mobile app or SPA directly calling an IdP). However, for a *confidential* client (a backend server with a secure `client_secret`, like our BFF talking to Google), Spring assumes the client secret provides sufficient protection against authorization code interception attacks and disables PKCE by default.

**Is forcing PKCE on a confidential client standard practice?**
Yes. It is defensible and increasingly recommended by modern OAuth 2.0 Security Best Current Practice (BCP) drafts. Even if the client secret protects the token exchange, PKCE provides defense-in-depth against code injection attacks if the authorization response (Step 4a in our trace) is intercepted or manipulated before reaching the BFF. We force it via a custom resolver because we prioritize defense-in-depth over Spring's permissive default.
