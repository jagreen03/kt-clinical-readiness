## Critical Gap 1: application.yml reconstruction drifted from canonical

Diff against the canonical Pass 3 file (which was direct-saved by Pilot at end of Pass 3):

| Property | Canonical | Pass 5 | Status |
| -------- | --------- | ------ | ------ |
| server.servlet.session.cookie.name | BFF_SESSION | SESSION | regressed |

| server.error.include-binding-errors | absent | never | added (defensible) |
| spring.security.oauth2.client.registration.google.redirect-uri | {baseUrl}/{action}/oauth2/code/{registrationId} | {baseUrl}/login/oauth2/code/{registrationId} | regressed |
| management.endpoints.web.exposure.include | health | health,info | loosened |
| management.endpoint.health.show-details | never | always | regressed (security inversion) |
| logging.level.org.springframework.security | INFO | absent | regressed |
| logging.level.org.springframework.security.oauth2 | INFO | DEBUG | regressed (production logging) |
| logging.level.com.bluesand.riverhorse | absent in base, DEBUG in dev | DEBUG in base | regressed (dev leaked to base) |

Most serious: show-details: always inverts the deliberate hardening from A-#1, where the rationale was that production health endpoints leak infrastructure metrics (database pool stats, disk usage). The inversion is invisible until something probes /actuator/health and gets back internal state.

Second serious: redirect-uri loses the {action} placeholder, which is the Spring template token resolving to "login" during the auth flow. Hardcoding "login" works today but loses framework idiom and breaks if Spring ever changes the action value.

Third: DEBUG-level logging on org.springframework.security.oauth2 in the base profile leaks OAuth2 token exchange artifacts and potential PII into production logs.

## Critical Gap 2: application-test.yml is non-hermetic

The Pass 5 stub:

  spring:
    security:
      oauth2:
        client:
          registration:
            google:
              client-id: test-client-id
              client-secret: test-client-secret

Missing: the provider.google block. Without it, Spring's OAuth2 client autoconfig calls Google's .well-known/openid-configuration discovery endpoint during context load. Tests must be hermetic - no network dependencies for unit-level smoke tests.

Resolution: two complementary fixes applied to the corrected file:

1. Add the provider.google stub block (issuer-uri, user-name-attribute).
2. Add spring.autoconfigure.exclude for OAuth2ClientAutoConfiguration to prevent discovery call entirely.

The combination is belt-and-suspenders. The exclude prevents the discovery call. The stub provider block keeps the bean shapes intact for any partial-context tests that need them.
