## Critical Gap 1: application.yml regenerated rather than transcribed

The Pass 3 prompt was explicit: "Do not regenerate the YAML content from scratch. Source from the KTS-0000006 A-#1 file. Only modify if Spring Boot 4 requires a property rename." Cynthia regenerated, and the following sections present in the A-#1 canonical source were dropped:

- server.servlet.session.timeout: 30m
- server.servlet.session.cookie.name: BFF_SESSION
- server.error.* (include-stacktrace, include-message, include-exception)
- spring.security.oauth2.client.registration.google.redirect-uri (explicit Path B declaration)
- spring.security.oauth2.client.registration.google.authorization-grant-type
- spring.security.oauth2.client.provider.google.user-name-attribute: sub
- management.endpoint.health.show-details: never
- logging.level.root: INFO and explicit oauth2 logger level

These were deliberate Pilot decisions in A-#1 (hardened error defaults, explicit Path B documentation, stable canonical user ID via sub). Stripping them silently regressed the security posture and the documentation discipline.

## Critical Gap 2: Actuator exposure loosened without justification

A-#1 set management.endpoints.web.exposure.include: health only, with explicit rationale that env and configprops would leak GOOGLE_OAUTH_CLIENT_SECRET if exposed in production. Pass 3 expanded to health,info,prometheus.

Two issues:

1. info is low-leak and defensible to add but was not justified in the synthesis.
2. prometheus requires micrometer-registry-prometheus to be on the classpath. It is not in pom.xml. The endpoint will silently 404 or be auto-configured as a no-op.

Resolution: revert to health-only in the base profile. Add info,env,beans,configprops to the dev profile only.

## Critical Gap 3: dev profile missing explicit on-profile declaration

Pass 3 application-dev.yml did not declare spring.config.activate.on-profile: dev. Spring Boot's filename-based profile activation makes this functionally optional, but A-#1 included the declaration to document intent. Cynthia dropped it.

## Minor: Spring Boot 4 property verification asserted, not demonstrated

The synthesis states the four property paths are stable in 4.0 but provides no evidence. A senior-grade answer cites the verification mechanism (Spring Boot 4 reference docs URL, mvn compile against 4.0.5 produces no deprecation warnings, etc.). Worth a documentation upgrade in future passes when verifying framework-version deltas.
