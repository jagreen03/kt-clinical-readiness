# KTS-0000008 Pass 4: Proxy Demo Component

## Decisions & Architecture Notes

* **Real Component vs. Inline Placeholders**: Upgrading `ProxyDemoComponent` from an inline placeholder to a standalone file allows us to inject the real `HttpClient` and `AuthService`, demonstrating true end-to-end reactivity and HTTP interceptor integration without cluttering the routing skeleton.
* **Component State Management**: Implemented the requested signal-based tri-state pattern for the response payload:
  - `undefined`: Initial state, no request sent.
  - `null`: Request completed with an error.
  - `value` (unknown type): Request succeeded, payload ready for rendering.
  - A separate `isLoading` signal handles the disabled state and loading text to keep the view clean.
* **Service ID Coupling**: The target URL `/api/proxy/clinical/sample` directly couples the SPA to the `proxy.services.clinical` route mapped in the BFF's `application.yml`. The path segment `/sample` is transparently appended to the downstream request by the BFF.
* **Expected Downstream Failure**: Since the BFF's `clinical` service URL points to a placeholder (`https://clinical-internal.example.com`), clicking the "Fetch" button will result in the Spring Boot proxy attempting to resolve and connect to that dummy domain. This will fail, prompting the BFF to return a 5xx gateway error. The SPA's `errorInterceptor` will catch and log this, and the component's local `catchError` subscription will update the signal state to `null` and render the error message on screen.
* **Carry-Forward Context**: The global HTTP error interceptor implemented in Pass 3 handles the console logging and any 401 redirects. The component itself only needs to handle local UI state reflection (showing the error message to the user) without reinventing global routing logic.
