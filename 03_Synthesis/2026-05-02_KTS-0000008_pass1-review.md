## Architecture Note 1: Custom CSRF interceptor exists because of cross-origin dev setup

Cynthia's synthesis claims the custom CSRF interceptor exists because Angular silently drops X-XSRF-TOKEN on cross-origin requests. The behavior description is accurate, but the framing matters: Angular drops the header cross-origin AS A SECURITY FEATURE - it prevents a malicious site from tricking your browser into sending your CSRF token to an attacker's domain.

The cleaner solution is proxy.conf.json (tunneling /api, /auth, /oauth2 through the dev server so the SPA appears same-origin to the browser). With the proxy in place, Angular's built-in XSRF handling works without a custom interceptor.

In production with the SPA served from the BFF, everything is same-origin and the built-in handling works without the custom interceptor.

So the custom interceptor is solving a problem that proxy.conf.json solves more cleanly. Carry-forward to Pass 5: add proxy.conf.json, document the same-origin model, consider removing the custom CSRF interceptor at that point.

For Pass 1, the interceptor works as written. Keep it.

## Architecture Note 2: provideClientHydration is unnecessary

provideClientHydration(withEventReplay()) is enabled in app.config.ts. The BFF is a Spring Boot app; it does not do Angular SSR. Hydration is a no-op or a confusion vector when the SPA is served as static files from the BFF or from ng serve.

Recommendation: drop provideClientHydration entirely. If SSR is ever introduced (e.g., Angular Universal in front of the BFF), the provider goes back in then.

Not a Pass 1 blocker. Defer to Pass 5 cleanup or technical debt saga.

## Verification Item: HttpXsrfTokenExtractor injection in functional interceptor

inject(HttpXsrfTokenExtractor) in a functional interceptor fires on every HTTP request. Functional interceptors run in injection context so the call works. Performance is fine (registry lookup, not new instance). If flaky CSRF behavior surfaces in tests or under load, this is the place to investigate. Not a known issue, just a forward note.

## Minor: CSRF interceptor missing OPTIONS exclusion comment

Lowercase comparison ['post', 'put', 'patch', 'delete'] correctly excludes OPTIONS, but only because OPTIONS is not in the list. Defensive: add a comment explaining why OPTIONS is excluded (preflight requests should never carry CSRF tokens; preflight is browser CORS check, not a real request). Prevents a future developer from adding OPTIONS "for completeness."

## Forward Note for Pass 3

withFetch() is enabled. Fetch error semantics differ slightly from XHR - fetch throws on network failures the same as on HTTP errors, and Angular wraps both into HttpErrorResponse. Pass 3 error interceptor will need to handle both classes (network vs HTTP) cleanly. Not a Pass 1 issue.

Pass 1 accepted. No code revisions required. Two architecture notes carry forward to Pass 5 (proxy.conf.json, drop hydration). One verification item logged. One minor cosmetic suggestion (OPTIONS comment) - defer or skip.
