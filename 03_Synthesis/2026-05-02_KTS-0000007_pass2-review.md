## Critical Gap 1: Code verifier generation is cryptographically weak

Cynthia regressed from the A-#2 implementation. Pass 2 source:

    String codeVerifier = Base64.getUrlEncoder().withoutPadding().encodeToString(
        UUID.randomUUID().toString().getBytes(StandardCharsets.US_ASCII)
    );

Three problems:

1. UUID.randomUUID() is the wrong entropy source. Version and variant nibbles are fixed bits, dashes are non-random structure. A 36-character UUID string has roughly 122 bits of effective entropy.
2. Re-encoding an already-ASCII UUID string as base64url adds length without adding entropy.
3. A-#2's original implementation was correct: Base64StringKeyGenerator with 96 raw random bytes from SecureRandom yields ~128 base64url characters with the full 768 bits of entropy.

Resolution: restore the A-#2 implementation. Replace the Pass 2 verifier line with:

    StringKeyGenerator secureKeyGenerator = new Base64StringKeyGenerator(
        Base64.getUrlEncoder().withoutPadding(), 96);
    String codeVerifier = secureKeyGenerator.generateKey();

Imports needed:

    import org.springframework.security.crypto.keygen.Base64StringKeyGenerator;
    import org.springframework.security.crypto.keygen.StringKeyGenerator;

The keygen reuses SecureRandom under the hood. This is the canonical Spring idiom for OAuth2 client secret material.

## Critical Gap 2: CSRF cookie configuration regressed

KTS-0000006 baseline included a CsrfTokenRequestAttributeHandler that Pass 2 silently dropped:

    Pass 2 (incomplete):
    .csrf(csrf -> csrf
        .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())
    )

    Baseline (correct):
    .csrf(csrf -> csrf
        .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())
        .csrfTokenRequestHandler(new CsrfTokenRequestAttributeHandler())
    )

In Spring Security 6.x and 7.x, CookieCsrfTokenRepository.withHttpOnlyFalse produces a deferred token by default. The token is generated lazily on first read. With Angular HttpClient, this causes the first XHR from the SPA to fail because XSRF-TOKEN has not yet materialized as a cookie. The CsrfTokenRequestAttributeHandler forces eager resolution.

Resolution: restore the .csrfTokenRequestHandler line. Add the import:

    import org.springframework.security.web.csrf.CsrfTokenRequestAttributeHandler;

## Verification: SecurityFilterChain bean parameter ordering

The signature injects pkceResolver as a method parameter:

    public SecurityFilterChain securityFilterChain(
        HttpSecurity http,
        OAuth2AuthorizationRequestResolver pkceResolver
    ) throws Exception

The pkceResolver bean depends on ClientRegistrationRepository, which is auto-configured by Spring's OAuth2 client autoconfiguration. With Spring Boot 4 and Spring Security 7's tightened bean lifecycle rules, this can race in edge cases. Not necessarily a bug; worth confirming with mvn test once Pass 3 application.yml lands and the OAuth2 client registration is real.

If the test reveals a bean resolution issue, two options:

1. Move the pkceResolver construction inline inside the filter chain method (don't expose as bean).
2. Add @Lazy to the pkceResolver parameter to defer resolution.

Defer the decision until Pass 3 / Pass 5 testing.

## Minor 1: spring-session-core orphaned bean risk

Adding spring-session-core gives the CookieSerializer / DefaultCookieSerializer API surface for compilation. Whether the cookie serializer bean is actually applied at runtime depends on whether a Spring Session store is configured.

The Pass 3 application.yml (drafted in KTS-0000006 A-#1) has spring.session.store-type: none, which effectively disables Spring Session. The CookieSerializer bean is orphaned at runtime; the SameSite=Lax setting does not reach the actual session cookie.

Two paths:

A. Upgrade to a real Spring Session store (in-memory MapSession, Redis, JDBC) so the CookieSerializer is wired. This requires a Pass 3 application.yml update.

B. Accept the orphan. SameSite hardening shifts to a different mechanism. Spring Boot 3.x and 4.x respect server.servlet.session.cookie.same-site directly, which is set in the Pass 3 yaml as same-site: lax. The CookieSerializer becomes redundant.

Recommendation: Path B for portfolio scope. Drop the CookieSerializer bean in a future revision when Pass 3 confirms server.servlet.session.cookie.same-site is the active mechanism. The bean does no harm, but the doctrine of one mechanism per concern is cleaner.

Not a blocker for Pass 2.

## Minor 2: Authentication failure handler not wired

No explicit failure handler on oauth2Login. Default behavior redirects to /login?error. For a BFF serving a SPA, that route does not exist.

Resolution: add a JSON-returning failure handler in Pass 3 or Pass 4. Pattern:

    .oauth2Login(oauth2 -> oauth2
        .authorizationEndpoint(...)
        .failureHandler((request, response, exception) -> {
            response.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
            response.setContentType("application/json");
            response.getWriter().write("{\"error\":\"authentication_failed\"}");
        })
    )

Defer to Pass 4 (controllers and routing) since failure handler logic naturally lives near the controller layer for SPA-friendly error shapes.
