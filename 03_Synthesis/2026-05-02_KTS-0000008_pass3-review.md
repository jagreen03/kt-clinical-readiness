## Critical Gap: 401 race between interceptor and AuthService.me()

When the route guard calls me() and the BFF returns 401:

1. Error interceptor fires first (closest to backend). Sets currentUser to null, navigates to '/'.
2. AuthService.me() catchError fires second. Sets currentUser to null again, emits of(null).
3. Guard's map(user => ...) runs on the of(null), returns UrlTree to '/' - but navigation already in progress.

Net result: duplicate state mutation, duplicate navigation. Angular cancels the redundant navigation, but the race may show as a UI flash in some browsers.

The same race exists for /auth/logout: if the interceptor sees 401 on logout (already-stale session), it would clear state AND the logout's finalize would clear state.

Resolution: scope the interceptor's 401 redirect logic to exclude URLs the auth service already owns:

  if (error.status === 401 && !req.url.match(/\/(api\/me|auth\/logout)/)) {
    authService.currentUser.set(null);
    router.navigate(['/']);
  }

For /api/me and /auth/logout, the AuthService methods own their own auth state lifecycle. The interceptor handles 401 globally for everything else.

## Gap 2 (Resolved): inFlight deduplication

Current code:

  this.inFlight = this.http.get<User>(...).pipe(
    tap(...),
    catchError(...),
    finalize(() => { this.inFlight = null; }),
    shareReplay(1)
  );

shareReplay(1) caches the last emitted value, but finalize runs on completion and immediately invalidates the cache via inFlight = null. So shareReplay(1) provides no benefit for sequential callers.

The intent (deduplicate concurrent guard activations) works correctly because both subscribers attach to the same observable while it is in-flight. The shareReplay is benign but redundant. Could simplify by dropping it, but current code is functionally correct. No fix required.

## Gap 3 (Folded into Gap 1 fix): /auth/logout 401 race

Same pattern as /api/me. The Gap 1 URL exclusion regex covers both endpoints.

## Minor 2: status === 0 network heuristic limitations

Catches CORS errors, unreachable host, request abort - all collapse to status 0 with no differentiation. Fine for portfolio scope. Production would need richer detection (timeout vs DNS vs CORS vs offline).

## Minor 3: 4xx other than 401/403 fall through to throwError

Intentional by design. Domain-specific 4xx errors are caller concerns. The interceptor surfaces them via throwError; callers handle 400/404/409/422 in their own catchError blocks. Confirmed correct.
